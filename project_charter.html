Problem Statement

Regional food supply chains operate in silos. Farms, quality inspection agencies, distributors, and retailers each maintain independent systems with no shared coordination layer. The result is duplicated effort, opaque status tracking, and slow response to quality or supply issues that cross organizational boundaries.

This project solves this challenge by delivering a unified digital platform that tracks produce from planting through to retail stocking — with every status change and cross-enterprise handoff visible in real time to all authorized stakeholders.

Project Purpose & Vision

To design and implement a Regional Farm-to-Table Food Network — a multi-enterprise communication ecosystem demonstrating that cooperative effort across independent enterprises creates greater value than any single entity operating alone.

The platform enables a complete produce journey: Plant → Harvest → Inspect → Certify → Warehouse → Order → Deliver → Stock — with every step tracked, status-managed, and visible across all four enterprises through a role-based Java Swing desktop application.

Business Justification

Efficiency: Eliminating manual handoffs between enterprises reduces coordination overhead and accelerates produce delivery timelines.

Transparency: Real-time status visibility across the supply chain enables faster issue resolution — inspection failures, delivery delays, and stock shortages are surfaced immediately.

Scalability: The ecosystem architecture (Ecosystem → Network → Enterprise → Organization) is designed to accommodate additional enterprises without restructuring the core platform.

Compliance: Certification tracking ensures only inspected and approved produce enters the distribution and retail pipeline.

Solution Architecture

Single-tier Java Swing desktop application. All state is held in-memory via Directory objects (ArrayList-backed) initialized at startup by ConfigureABusiness using the java-faker library for realistic seed data. The system follows a strict containment hierarchy and role-based authentication pattern.

In Scope

  • ✓Java Swing desktop application — single deployable JAR
  • ✓Role-based authentication — 12 profiles routed via instanceof pattern
  • ✓4 enterprise domain models with full CRUD operations
  • ✓6 work request types — 3 cross-enterprise, 3 cross-organization
  • ✓14+ UI panels following consistent UIFactory design system
  • ✓Status management via StatusConstants — no inline strings
  • ✓Pre-populated seed data via java-faker library
  • ✓NetworkSummary reporting — KPI dashboard for Quality Analyst
  • ✓System Admin CRUD — full account management
  • ✓Enterprise Admin CRUD — scoped by enterpriseId
  • ✓Form validation — required fields, email format, date regex, positive integers
  • ✓Auto-generated unique IDs via static counters on all domain classes
  • ✓UML class diagrams, component diagram, PM documentation package

Out of Scope

  • ✗Database persistence — all state is in-memory, resets on restart
  • ✗Multi-user concurrent sessions — single JVM, single user at a time
  • ✗Password encryption — plain string storage within demo scope
  • ✗Real-time clock or scheduling — dates entered manually as strings
  • ✗Email / SMS notifications — no external API integrations
  • ✗Map or geolocation features
  • ✗Partial delivery handling — PurchaseOrders fulfilled in full only
  • ✗Re-inspection loops — failed inspections are terminal
  • ✗Back-flow on certification denial — denied certs are terminal
  • ✗Mobile or web interface — desktop Swing only
  • ✗Reporting export — analytics visible on-screen only

Key Design Decisions & Constraints

DecisionResolutionRationale
Status value formatPlain String constants in StatusConstants.java — no enumsSimpler refactoring, consistent across all enterprise boundaries
Cross-enterprise sender / receiverSending side creates the request and sets initial status. Receiving side only calls updateStatus()Clear ownership, prevents duplicate object creation
Quantity type boundaryHarvestBatch.quantityKg = float (raw weight). All downstream quantities = int (packaged units)Reflects real-world handoff from raw produce to packaged goods
PurchaseOrder ↔ WarehouseItem couplingPurchaseOrder stores product name (String) + qty (int) only — no WarehouseItem referenceKeeps Retail and Distribution enterprises fully decoupled at the model level
Inspector / Driver assignmentSelf-assign via Claim button — no admin assignment panelReduces UI complexity, mirrors real-world field worker claim patterns
ConfigureABusiness org referencesAlways use static fields — never new Organization()Ensures all WorkRequests reference the same org objects, enabling cross-enterprise filtering

Agile / Scrum Framework

This project is delivered using a Scrum-inspired framework with 5 time-boxed sprints. Each sprint has a defined goal, a backlog of work packages drawn from the WBS, a sprint review milestone, and a retrospective. Parallel enterprise tracks in Sprint 2 reflect Scrum's cross-functional team model — each enterprise lead operates as an independent squad delivering their increment simultaneously.

Sync points between enterprises are treated as inter-team dependencies and managed via the dependency log. Any sync point at risk triggers an immediate blocker escalation per the communication plan.
